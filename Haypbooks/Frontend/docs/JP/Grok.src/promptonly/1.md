**Developer Notes & Implementation Details**

- **API Endpoints**
  - Server route: `POST /auth/pin/setup` — body `{ pin, pinConfirm }` → saves hashed PIN; responds `{ success: true, pinSetAt: "2025-12-24T12:34:56.789Z" }` on success (useful for client UX). In the frontend (dev/prod) we call `/api/auth/pin/setup` and rely on the proxy / reverse-proxy to route `/api/*` to the backend.
  - Server route: `POST /auth/pin/verify` — body `{ pin }` → returns `{ success: true }` when valid; respond 401/false when invalid. Frontend: `/api/auth/pin/verify`.
  - Server route: `POST /auth/send-verification` — body `{ email }` → sends OTP; in dev returns `{ success: true, otp }` for testing (do NOT return OTP in prod). Frontend: `/api/auth/send-verification`.
  - Server route: `POST /auth/verify-otp` — body `{ email, otpCode }` → verifies OTP and returns `{ success: true }`. Frontend: `/api/auth/verify-otp`.
  - Server route: `GET /users/me` — returns current user data including `hasPin` (boolean) and `pinSetAt` (ISO timestamp) so the client can deterministically show PIN options. Frontend: `/api/users/me` (proxied).

  Example `GET /api/auth/me` response:

  ```json
  {
    "user": { "id": "u_123", "email": "you@example.com", "hasPin": false, "pinSetAt": null }
  }
  ```

  Example partial `POST /api/auth/login` response when setup is required:

  ```json
  {
    "user": { "id": "u_123", "email": "you@example.com", "requiresPinSetup": true }
  }
  ```

  Using these explicit flags avoids client-side inference and makes UI flows deterministic.

- **Dev testing & helpers**
  - Signup currently returns `_devOtp` in non-production builds for quick testing; there is a test helper endpoint `/api/test/otp/latest?email=&purpose=VERIFY` to retrieve the last OTP for a user in dev. Note: test endpoints are enabled when `ALLOW_TEST_ENDPOINTS=true` is set in the backend environment (used in CI and local e2e runs).
  - Use these only in development and never expose them in production.

- **Security & operational notes**
  - PINs must be stored as salted hashes (bcrypt/Argon2); do not store plaintext.
  - OTP codes should expire after a short TTL (recommended: **10 minutes (600 seconds)**) and be rate-limited on send to prevent abuse (e.g., 5 sends per 60 minutes). Make sure the email sample and API docs use the same TTL value to avoid client/UX confusion.
  - Record a security/audit event on PIN setup and PIN reset actions.

- **Reset PIN flow (recommended)**
  - Start with an explicit "Forgot PIN? Reset PIN" CTA on the PIN entry screen.
  - Send OTP to the user's email (`POST /api/auth/send-verification`) and verify via `POST /api/auth/verify-otp`.
  - On successful OTP verification, allow the user to choose a new PIN (validate & save hashed PIN and update `pinSetAt`).
