# 🏠 HOME Module

## 1. Dashboard

### 1. Page Overview
- **Page Name:** Dashboard  
- **Goal:** Provide a real-time snapshot of business health, key metrics, and actionable items for the user. Serve as the primary landing page that guides users to critical tasks and highlights areas requiring attention.  
- **Accounting Impact:** Displays aggregated financial data (cash position, receivables, payables, revenue, expenses) that reflect the current state of the general ledger. Does not modify accounting records but surfaces critical accounting-related alerts and metrics that may prompt users to take accounting actions such as approvals, reconciliations, or period closes.

---

### 2. Tables & Data Structures

- **Main Table(s):**
  - `Workspace` — Provides workspace context and subscription status.
  - `Company` — Provides company-level settings and currency information.
  - `AccountBalance` — Stores month-end balances for display in trend charts.
  - `Invoice` — Aggregated for AR aging snapshot.
  - `Bill` — Aggregated for AP aging snapshot.
  - `BankAccount` — Aggregated for cash position display.
  - `Task` — For displaying user's assigned and overdue tasks.
  - `JournalEntry` — For recent activity feed.
  - `Notification` — For notification center preview.
  - `KpiDashboard` / `DashboardWidget` — For customizable dashboard layouts.
  - `AiInsight` — For AI-generated insights and recommendations.

- **Relationships:**
  - `Workspace` → `Company` (one-to-many)
  - `Company` → `AccountBalance` (one-to-many)
  - `Company` → `Invoice` (one-to-many)
  - `Company` → `Bill` (one-to-many)
  - `Company` → `BankAccount` (one-to-many)
  - `WorkspaceUser` → `Task` (assigned tasks)
  - `User` → `Notification` (user-specific notifications)

- **Schema Mapping:**
  - Cash Position: `SELECT SUM(balance) FROM BankAccount WHERE workspaceId = ? AND isDefault = true`
  - AR Aging: `SELECT SUM(balance) FROM Invoice WHERE companyId = ? AND status IN ('SENT', 'PARTIAL', 'OVERDUE') GROUP BY dueDate`
  - AP Aging: `SELECT SUM(balance) FROM Bill WHERE companyId = ? AND status IN ('APPROVED', 'OVERDUE') GROUP BY dueAt`
  - Account Balances: `AccountBalance` table keyed by `companyId`, `accountId`, `yearMonth`

---

### 3. Filters & Search

- **Basic Filters:**
  - **Date Range Selector:** Allows filtering metrics by `Today`, `This Week`, `This Month`, `This Quarter`, `This Year`, or custom date range. Affects all displayed metrics and charts.
  - **Company/Entity Selector:** Multi-company workspaces can filter the dashboard to show data for a specific company or all companies.
  - **Currency Display:** Option to view amounts in functional currency or base currency for consolidated views.

- **Advanced Filtering:**
  - **Metric Configuration:** Users can select which KPIs to display (e.g., toggle off inventory metrics if not applicable).
  - **Comparison Period:** Compare current period vs. prior period or budget.
  - **Branch/Location Filter:** Filter data by `Location` if multi-location is enabled.
  - **Department/Class Filter:** Filter by `Class` for departmental dashboards.

- **Purpose:** Filters enable executives to focus on specific timeframes, business units, or metrics relevant to their decision-making. Date range filtering is essential for period-over-period comparisons and trend analysis, while entity filtering supports consolidated vs. entity-specific views critical for multi-entity accounting.

---

---

### 5. User Roles & Permissions

- **Who can access:** All authenticated users with at least `Viewer` role on the workspace.
- **Restrictions:**
  - **Viewers:** Can only view dashboard; cannot customize layout or dismiss insights.
  - **Members:** Can customize their personal dashboard layout; cannot access company-wide settings.
  - **Admins/Owners:** Can customize shared dashboard layouts for all users; can configure which metrics are visible to different roles.
  - **Accountants:** Have access to all financial metrics including sensitive profitability data.
  - **Restricted Users:** May see limited metrics based on department/location restrictions configured in `EntityDimensionValue`.

---

### 6. Edge Cases & Validation

- **Business Rules:**
  - Dashboard must handle companies with no transaction history gracefully (show zero balances, empty charts).
  - Multi-currency companies must display amounts consistently (either converted to base currency or shown in original currency with conversion indicator).
  - Widgets that require specific modules (e.g., Inventory) must be hidden if the module is not enabled for the company (`Company.inventoryEnabled`).

- **Validation Checks:**
  - Ensure date range filters don't exceed system limits (e.g., cannot view data before company start date).
  - Verify user has permission to view metrics for selected company/location.
  - Handle null or missing data by showing "N/A" or "No data" indicators rather than breaking the UI.
  - Rate limit refresh operations to prevent excessive database queries.

## 2. Business Health

### 1. Page Overview
- **Page Name:** Business Health  
- **Goal:** Provide a comprehensive, visual assessment of the company's financial health through key performance indicators, ratios, and trend analysis. Enable owners and executives to quickly identify areas of strength and concern.  
- **Accounting Impact:** Displays calculated financial ratios and metrics derived from GL data. Does not modify accounting records but provides visibility into the results of accounting activities and may prompt users to investigate or take corrective actions.

---

### 2. Tables & Data Structures

- **Main Table(s):**
  - `FinancialRatio` — Stores calculated ratios (current ratio, quick ratio, debt-to-equity, etc.).
  - `FinancialMetric` — Stores time-series metrics (AR turnover, AP turnover, etc.).
  - `AccountBalance` — Source data for ratio calculations.
  - `CashFlowForecast` — For liquidity projections.
  - `CreditHealthScore` — For creditworthiness assessment.
  - `AiInsight` — AI-generated health assessments and recommendations.

- **Relationships:**
  - `Company` → `FinancialRatio` (one-to-many by period)
  - `Company` → `FinancialMetric` (one-to-many by period)
  - `Company` → `AccountBalance` (one-to-many)
  - `Company` → `CashFlowForecast` (one-to-many)
  - `Company` → `CreditHealthScore` (one-to-many)

- **Schema Mapping:**
  - Current Ratio = Total Current Assets / Total Current Liabilities (from `AccountBalance`)
  - Quick Ratio = (Cash + Accounts Receivable) / Current Liabilities
  - Debt-to-Equity = Total Liabilities / Total Equity
  - AR Turnover = Net Credit Sales / Average AR
  - AP Turnover = Net Credit Purchases / Average AP

---

### 3. Filters & Search

- **Basic Filters:**
  - **Period Selector:** View health metrics for specific month/quarter/year.
  - **Comparison Toggle:** Compare against industry benchmarks, prior periods, or budget.

- **Advanced Filtering:**
  - **Metric Categories:** Filter by Liquidity, Profitability, Efficiency, Solvency categories.
  - **Threshold Alerts:** Show only metrics that fall outside acceptable ranges (red/yellow status).
  - **Benchmark Source:** Select industry benchmarks for comparison.

- **Purpose:** Enables targeted analysis of financial health dimensions. Period filtering is essential for trend analysis and identifying deterioration or improvement over time, which supports quarterly reviews and strategic planning decisions.

---

### 4. Buttons & Actions

| Button Name | Purpose | Effect in Accounting | Data Flow |
|-------------|---------|---------------------|-----------|
| **Recalculate Ratios** | Forces recalculation of all ratios for the selected period | No accounting effect; updates `FinancialRatio` and `FinancialMetric` records | Triggers calculation jobs |
| **Export Report** | Generates a PDF/Excel report of business health metrics | No accounting effect | Creates export file |
| **Set Alert Threshold** | Configures alerts when metrics fall outside specified ranges | No accounting effect | Updates user preferences |
| **View Detailed Calculation** | Shows the breakdown of how each ratio was calculated | No accounting effect | Displays modal with calculation details |
| **Drill Down** | Navigates to the source GL accounts or transactions for a metric | No accounting effect | Navigation to GL or reports |

---

### 5. User Roles & Permissions

- **Who can access:** Owners, Admins, Accountants, and users with `View Financial Reports` permission.
- **Restrictions:**
  - Non-financial users may see limited metrics (e.g., only operational metrics, not profitability).
  - Benchmarks may be available only to premium subscription tiers.
  - Credit health scores may be restricted to owners and finance team only.

---

### 6. Edge Cases & Validation

- **Business Rules:**
  - Ratios must handle division by zero gracefully (e.g., if current liabilities = 0, current ratio shows "N/A").
  - Negative equity situations must be flagged with appropriate warnings.
  - Trend arrows must account for whether an increase is positive or negative (e.g., increasing debt-to-equity is negative, increasing current ratio is positive).

- **Validation Checks:**
  - Ensure period has sufficient data for meaningful calculations.
  - Warn if comparing periods with different chart of accounts structures.
  - Validate that benchmark data is available for the selected industry.

# ✅ TASKS & APPROVALS Module

## 1. My Tasks

### 1. Page Overview
- **Page Name:** My Tasks  
- **Goal:** Display all tasks assigned to the current user, enabling efficient task management and tracking. Serve as the primary work queue for individual contributors.  
- **Accounting Impact:** Tasks may be linked to accounting entities (invoices, bills, journal entries, reconciliations). Completing accounting-related tasks often results in accounting actions being taken, but the task system itself does not modify accounting records.

---

### 2. Tables & Data Structures

- **Main Table(s):**
  - `Task` — Core task entity with status, priority, due date, and relations.
  - `WorkspaceUser` — The user's workspace membership record.
  - `TaskComment` — Comments on tasks for collaboration.
  - `User` — Task creator and assignee information.

- **Relationships:**
  - `Task.assigneeId` → `WorkspaceUser.userId` (the assigned user)
  - `Task.createdById` → `User.id` (the task creator)
  - `Task.workspaceId` → `Workspace.id`
  - `Task.companyId` → `Company.id` (optional, for company-scoped tasks)
  - `Task.practiceId` → `Practice.id` (optional, for practice-scoped tasks)
  - `Task.relatedType` and `Task.relatedId` — Polymorphic relation to entities like Invoice, Bill, JournalEntry

- **Schema Mapping:**
  ```sql
  SELECT * FROM Task 
  WHERE workspaceId = ? 
  AND assigneeId = ? 
  AND status NOT IN ('DONE', 'CANCELLED')
  ORDER BY 
    CASE priority 
      WHEN 'URGENT' THEN 1 
      WHEN 'HIGH' THEN 2 
      WHEN 'MEDIUM' THEN 3 
      WHEN 'LOW' THEN 4 
    END,
    dueDate ASC NULLS LAST
  ```

---

### 3. Filters & Search

- **Basic Filters:**
  - **Status:** Pending, In Progress, Blocked, Done, Cancelled.
  - **Priority:** Urgent, High, Medium, Low.
  - **Due Date:** Overdue, Due Today, Due This Week, Due This Month, Custom Range.
  - **Search:** Full-text search on title and description.

- **Advanced Filtering:**
  - **Related Entity Type:** Filter by tasks linked to specific entity types (Invoice, Bill, Project, etc.).
  - **Created By:** Filter by who created the task.
  - **Created Date Range:** Filter by task creation date.
  - **Has Comments:** Show only tasks with comments.
  - **Has Attachments:** Show only tasks with attachments.

- **Purpose:** Enables users to focus on the most urgent and relevant tasks. Priority and due date filtering are essential for time management, while entity type filtering helps users working on specific accounting workflows (e.g., all invoice-related tasks).

---

### 4. Buttons & Actions

| Button Name | Purpose | Effect in Accounting | Data Flow |
|-------------|---------|---------------------|-----------|
| **New Task** | Creates a new task | None (unless linked to an entity, then creates relationship only) | Creates `Task` record |
| **Mark Complete** | Marks a task as done | May trigger workflow automation; no direct accounting effect | Updates `Task.status` to 'DONE', sets `completedAt` |
| **Edit Task** | Modifies task details | None | Updates `Task` fields |
| **Delete Task** | Removes a task | None | Soft deletes by setting `deletedAt` |
| **Add Comment** | Adds a comment to a task | None | Creates `TaskComment` record |
| **Set Reminder** | Schedules a reminder for the task | None | Updates `Task.remindAt` |
| **Change Priority** | Updates task priority | None | Updates `Task.priority` |
| **Reassign** | Assigns task to another user | None | Updates `Task.assigneeId` |
| **View Related Entity** | Navigates to the linked entity (invoice, bill, etc.) | None | Navigation action |

---

### 5. User Roles & Permissions

- **Who can access:** All workspace members.
- **Restrictions:**
  - Users can only edit/reassign tasks assigned to them or created by them.
  - Users cannot delete tasks created by others unless they have `Admin` role.
  - Tasks in a company the user doesn't have access to are not visible.

---

### 6. Edge Cases & Validation

- **Business Rules:**
  - Tasks cannot have both `companyId` and `practiceId` set (XOR constraint).
  - Completed tasks cannot be edited except to add comments.
  - Reminders are automatically marked as sent (`reminderSent = true`) after the reminder is delivered.
  - Overdue tasks are automatically flagged in the UI.

- **Validation Checks:**
  - Title is required (max 200 characters).
  - Due date cannot be in the past for new tasks (warning only).
  - Assignee must be a member of the same workspace.
  - Related entity must exist and be accessible to the user.

## 2. My Approvals

### 1. Page Overview
- **Page Name:** My Approvals  
- **Goal:** Display all items pending the current user's approval (invoices, bills, journal entries, payments, expenses). Enable efficient review and approval/rejection workflow.  
- **Accounting Impact:** Approvals directly affect accounting records. Approving a transaction typically changes its status to "Approved" or "Posted," making it part of the official accounting records and affecting GL balances.

---

### 2. Tables & Data Structures

- **Main Table(s):**
  - `DocumentApproval` — Core approval tracking entity.
  - `Invoice` — For invoice approvals.
  - `Bill` — For bill approvals.
  - `JournalEntry` — For journal entry approvals.
  - `BillPayment` — For payment approvals.
  - `PaymentReceived` — For received payment approvals.
  - `ExpenseClaim` — For expense claim approvals.
  - `CustomerRefund` / `VendorRefund` — For refund approvals.
  - `RefundApproval` — Refund-specific approval tracking.
  - `InventoryAdjustmentRequest` — For inventory adjustment approvals.
  - `InventoryAdjustmentApproval` — Inventory approval tracking.

- **Relationships:**
  - `DocumentApproval.approverId` → `User.id`
  - `DocumentApproval.documentType` + `documentId` → Polymorphic relation to Invoice, Bill, etc.
  - `RefundApproval.approverId` → `User.id`
  - `InventoryAdjustmentApproval.approverId` → `User.id`

- **Schema Mapping:**
  ```sql
  SELECT * FROM DocumentApproval 
  WHERE approverId = ? 
  AND status = 'PENDING'
  ORDER BY createdAt DESC
  ```

---

### 3. Filters & Search

- **Basic Filters:**
  - **Document Type:** Invoice, Bill, Journal Entry, Payment, Expense Claim, Refund.
  - **Status:** Pending, Approved, Rejected.
  - **Date Range:** Filter by approval request date.
  - **Amount Range:** Filter by transaction amount.

- **Advanced Filtering:**
  - **Requester:** Filter by who submitted the item for approval.
  - **Vendor/Customer:** Filter by related vendor or customer.
  - **Priority Level:** Items exceeding certain thresholds may be flagged as high priority.
  - **Approval Level:** For multi-level approvals, filter by approval level.

- **Purpose:** Enables approvers to prioritize high-value or time-sensitive items. Document type filtering supports role-based approval workflows (e.g., AR manager approves invoices, AP manager approves bills).

---

### 4. Buttons & Actions

| Button Name | Purpose | Effect in Accounting | Data Flow |

| **Reject with Comments** | Rejects with reason | Same as Reject | Updates approval record with rejection reason |

---

### 5. User Roles & Permissions

- **Who can access:** Users who have been designated as approvers in approval workflows or matrices.
- **Restrictions:**
  - Users cannot approve their own submitted items (segregation of duties).
  - Approval limits may restrict maximum approval amounts based on role.
  - Multi-level approvals require sequential approval (Level 1 before Level 2).
  - Some document types may require specific role permissions (e.g., only accountants can approve journal entries).

---

### 6. Edge Cases & Validation

- **Business Rules:**
  - Approval thresholds (`ApprovalThreshold`) determine who can approve items of certain amounts.
  - If an approver is on leave, delegations (`Delegation Rules` in Automation) can auto-forward items.
  - Partial approvals are not supported for most document types (all-or-nothing approval).
  - Approval deadlines may be enforced with escalation rules.

- **Validation Checks:**
  - User must have approval authority for the document type.
  - Amount must be within user's approval limit.
  - For multi-level approvals, all previous levels must be approved.
  - Document must be in a submittable status (not already posted/voided).

## 4. Overdue Items

### 1. Page Overview
- **Page Name:** Overdue Items  
- **Goal:** Consolidate all overdue items across the system for the user's attention, including overdue tasks, invoices, bills, approvals, and compliance deadlines. Enable proactive management of aging items.  
- **Accounting Impact:** Overdue invoices and bills directly affect cash flow and financial statements. Resolving overdue items (collecting payments, paying bills) results in accounting transactions.

---

### 2. Tables & Data Structures

- **Main Table(s):**
  - `Task` — Overdue tasks (due date passed, status not DONE).
  - `Invoice` — Overdue invoices (status = 'OVERDUE').
  - `Bill` — Overdue bills (status = 'OVERDUE').
  - `TaxObligation` — Overdue tax obligations.
  - `ComplianceDeadline` — Overdue compliance deadlines.
  - `LocalTaxObligation` — Overdue local tax obligations.

- **Relationships:**
  - All tables link to `Company` via `companyId`.
  - Tasks link to assigned `User` via `assigneeId`.

- **Schema Mapping:**
  ```sql
  -- Overdue Tasks
  SELECT * FROM Task WHERE dueDate < NOW() AND status NOT IN ('DONE', 'CANCELLED')
  
  -- Overdue Invoices
  SELECT * FROM Invoice WHERE dueDate < NOW() AND status = 'OVERDUE' AND balance > 0
  
  -- Overdue Bills
  SELECT * FROM Bill WHERE dueAt < NOW() AND status = 'OVERDUE' AND balance > 0
  ```

---

### 3. Filters & Search

- **Basic Filters:**
  - **Item Type:** Tasks, Invoices, Bills, Tax Obligations, Compliance Deadlines.
  - **Days Overdue:** 1-7 days, 8-30 days, 31-60 days, 60+ days.
  - **Amount Range:** For monetary items.

- **Advanced Filtering:**
  - **Customer/Vendor:** Filter by related party.
  - **Priority:** Sort by impact (highest amount, oldest first).

- **Purpose:** Enables focused management of aged items. Days overdue filtering supports escalation workflows, while amount sorting helps prioritize high-impact items.

---

### 4. Buttons & Actions

| Button Name | Purpose | Effect in Accounting | Data Flow |

---

### 2. Tables & Data Structures

  - **Amount Range:** Payment amount.

- **Advanced Filtering:**
  - **Requester:** Filter by payment initiator.
  - **Vendor/Beneficiary:** Filter by recipient.
  - **Approval Level:** Filter by required approval level.

- **Purpose:** Enables payment control and authorization. Status filtering prioritizes pending approvals.

### 4. Buttons & Actions
