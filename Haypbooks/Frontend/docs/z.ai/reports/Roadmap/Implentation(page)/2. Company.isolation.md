# 6. Optional but highly recommended: Row-Level Security (RLS)

Even with perfect query filtering, bugs can happen. RLS enforces isolation **at the database level**, ensuring no data crosses tenant boundaries even if a query is written incorrectly.

### Why RLS is valuable

- Protects against accidental data leaks from developer mistakes
- Ensures isolation even for direct DB access (scripts, migrations, analytics)
- Provides a last line of defense in multi-tenant environments

### How to implement it (high level)

1. Enable RLS on the sensitive table(s) (e.g., `Account`, `JournalEntry`, `Invoice`).
2. Create a policy that allows access only when `companyId = current_setting('haypbooks.current_company')` (or similar).
3. Set the session variable (`SET haypbooks.current_company = '...'`) at auth time.

# 5. Recommended Prisma Safety Pattern

Create a **global Prisma middleware**.

Example:

```
prisma.$use(async (params, next) => {

  if (params.model && params.action === "findMany") {
      params.args.where = {
         ...params.args.where,
         companyId: currentCompanyId
      }
  }

  return next(params)

})
```

This automatically enforces isolation.
